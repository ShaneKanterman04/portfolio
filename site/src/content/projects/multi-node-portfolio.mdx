## Overview

The portfolio is intentionally static. The operational work is in making that small application deploy safely, remain private behind a shared edge, and recover without rebuilding a server by hand.

The current production path is **Cloudflare → kanter-edge → private portfolio LXC**. Cloudflare provides the public network and validates the origin certificate. Caddy owns public ingress on the shared edge VM. Nginx serves the generated Astro files from a dedicated unprivileged container.

## Current Architecture

```text
Internet
   |
Cloudflare DNS, CDN, Full (strict) TLS
   |
kanter-edge VM · Caddy
   |
Private Kantercloud network
   |
Unprivileged Debian LXC · Nginx · Astro files
```

## Why This Shape

The edge VM already owns the server's public ports and certificate lifecycle. Putting the portfolio there would mix ingress and workload state. Giving the portfolio LXC its own Tailscale node would duplicate network identity and administration.

The resulting boundary is small and explicit:

- the Proxmox host routes public 80/443 only to `kanter-edge`
- Caddy proxies the portfolio hostname over the private guest network
- the LXC accepts HTTP only from approved infrastructure addresses
- deployment reaches private SSH through the existing subnet router
- the LXC runs no Docker daemon and no Tailscale daemon

## Atomic Delivery

GitHub Actions builds and tests the Astro site on a hosted runner. The deployment job joins the tailnet as a short-lived tagged node and sends a compressed release through a restricted SSH key.

The remote account cannot open a shell. It can only request `deploy <sha>` or `rollback <sha>`. The deploy command validates the archive, extracts it to a temporary directory, moves it into a commit-addressed release, and swaps the `current` symlink atomically. If local HTTP validation fails, the previous symlink is restored.

Five releases stay on disk, so a rollback does not depend on GitHub, npm, or a rebuild.

## TLS and Health

Cloudflare uses **Full (strict)** origin mode and requires TLS 1.2 or newer from visitors. Caddy obtains and renews publicly trusted certificates for the apex and `www` hostnames. The `www` hostname redirects to the canonical apex URL.

Nginx exposes `/healthz` with an exact `ok` response. Cloudflare checks the route through the edge, while Homebase checks both the public site and the private origin. That separates an origin failure from a DNS, TLS, or edge-routing failure.

## Recovery

The deployment has three recovery layers:

| Failure | Recovery |
|---|---|
| Bad site release | Activate a retained commit SHA |
| Broken container | Restore the nightly Proxmox snapshot |
| Migration regression | Temporarily re-enable the retained GCP origin |

<Callout type="note">
  The rollback GCP origin is temporary. It remains disabled during the soak window and can be removed once the Kantercloud deployment has a clean operating history.
</Callout>

## Architecture Evolution

The first version used two GCP `e2-micro` origins behind a paid Cloudflare Load Balancer. That design was useful for learning Terraform, multi-zone origins, and health checks. Over time the second origin became disabled, leaving the published redundancy claim out of step with reality.

The migration kept the useful parts—static output, Nginx, Cloudflare, automated delivery—and replaced nominal redundancy with controls that are exercised directly: private origin access, atomic releases, verified TLS, explicit health checks, and snapshot recovery.

## What It Shows

- **Linux operations:** a hardened Debian service with narrow firewall rules
- **Network design:** public edge and private workload roles kept separate
- **Delivery engineering:** build once, validate, atomically activate, and retain rollback state
- **Operational honesty:** documentation and visible site claims match the running system
- **Cost judgment:** reuse existing reliable infrastructure instead of paying for an idle load-balancer design
